# Payload e Response

## Payload do Pedido (Request Body)

A Nexfar envia o seguinte payload JSON no body do POST. Todos os campos são autoexplicativos e seguem o formato padronizado Nexfar.

### Campos do Pedido (Header)

| Campo | Tipo | Obrigatório | Descrição |
|-------|------|-------------|-----------|
| `nexfarOrderId` | Long | Sim | ID interno Nexfar do pedido — **chave canônica de idempotência** |
| `orderDate` | String (ISO 8601) | Sim | Data do pedido no formato `yyyy-MM-dd'T'HH:mm:ss'Z'` |
| `branchId` | String | Sim | Identificador externo da filial (valor do `externalId` cadastrado no seu sistema) |
| `warehouseId` | String | Sim | Identificador externo do depósito/armazém (valor do `externalId` cadastrado no seu sistema) |
| `clientCnpj` | String | Sim | CNPJ do cliente (farmácia) |
| `clientName` | String | Sim | Nome/razão social do cliente |
| `clientId` | String | Sim | Identificador externo do cliente (valor do `externalId` cadastrado no seu sistema) |
| `sellerId` | String | Não | Identificador externo do vendedor (valor do `externalId` cadastrado no seu sistema) |
| `mainSellerId` | String | Não | Identificador externo do vendedor principal (valor do `externalId` cadastrado no seu sistema) |
| `userRole` | String | Não | Papel do usuário que criou o pedido (ex: `SELLER`, `MANAGER`) |
| `paymentMethod` | String | Não | Método de pagamento (ex: `BOLETO`, `CASH`, `CREDIT_CARD`) |
| `paymentExternalId` | String | Não | ID da condição de pagamento no sistema do distribuidor |
| `paymentMediumTerm` | Integer | Não | Prazo médio de pagamento em dias |
| `comments` | Object | Não | Comentários do vendedor, cliente e gerente de vendas |
| `comments.seller` | String | Não | Comentário do vendedor |
| `comments.client` | String | Não | Comentário do cliente |
| `comments.salesManager` | String | Não | Comentário do gerente de vendas |
| `originalTotal` | Decimal | Sim | Valor total do pedido (sem impostos) |
| `originalTotalWithTaxes` | Decimal | Sim | Valor total do pedido (com impostos) |
| `items` | Array | Sim | Lista de itens do pedido |

### Campos dos Itens

| Campo | Tipo | Obrigatório | Descrição |
|-------|------|-------------|-----------|
| `sku` | String | Sim | Código SKU do produto no distribuidor |
| `ean` | String | Sim | Código de barras EAN do produto |
| `productName` | String | Sim | Nome do produto |
| `quantityRequested` | Integer | Sim | Quantidade solicitada |
| `price` | Decimal | Sim | Preço unitário (sem impostos) |
| `finalPrice` | Decimal | Sim | Preço unitário final (com ST/impostos) |
| `subsidized` | Boolean | Não | Se o item é subsidiado |
| `promotionId` | Long | Não | ID da promoção aplicada (se houver) |
| `sellerKickback` | Decimal | Não | Comissão do vendedor sobre o item |

<Tip>
  **Datas** seguem o formato ISO 8601: `yyyy-MM-dd'T'HH:mm:ss'Z'` (UTC). Certifique-se de que seu parser aceita este formato.
</Tip>

## Response Esperada

O endpoint do distribuidor **deve retornar** um JSON com o seguinte formato:

| Campo | Tipo | Obrigatório | Descrição |
|-------|------|-------------|-----------|
| `orderExternalId` | String | **Sim** | ID do pedido no sistema do distribuidor |
| `status` | String | Não | Status do pedido no distribuidor (ex: `RECEIVED`, `PROCESSING`) |
| `message` | String | Não | Mensagem descritiva (ex: `Pedido recebido com sucesso`) |

### Exemplo de Response de Sucesso

```json
{
  "orderExternalId": "PED-2026-00458",
  "status": "RECEIVED",
  "message": "Pedido recebido e em processamento"
}
```

<Warning>
  O campo `orderExternalId` é **obrigatório** e deve ser preenchido. Se a response não contiver este campo (ou vier como `null`/vazio), a Nexfar tratará como falha e fará retry automático.
</Warning>

## Códigos HTTP

A Nexfar interpreta os códigos HTTP da seguinte forma:

| Status Code | Interpretação | Ação da Nexfar |
|---|---|---|
| `2xx` + `orderExternalId` preenchido | **Sucesso** | Pedido marcado como recebido pelo distribuidor |
| `2xx` + `orderExternalId` vazio/null | Falha recuperável | Retry automático |
| `4xx` | Falha recuperável | Retry automático |
| `5xx` | Falha recuperável | Retry automático |
| Timeout / conexão recusada | Falha recuperável | Retry automático |

<Note>
  Todos os cenários de falha geram **retry automático** até o limite configurado. Se seu endpoint retornar erro temporário (ex: `503 Service Unavailable`), a Nexfar tentará novamente automaticamente.
</Note>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Response sem orderExternalId">
    **Problema:** Seu endpoint retorna HTTP 200 mas sem o campo `orderExternalId` no JSON.

    **Solução:** Certifique-se de que a response sempre contém o campo `orderExternalId` preenchido com o ID do pedido no seu sistema. A Nexfar trata responses sem este campo como falha.

    ```json
    // Errado
    { "status": "OK" }

    // Correto
    { "orderExternalId": "PED-123", "status": "OK" }
    ```
  </Accordion>

  <Accordion title="Timeout na requisição">
    **Problema:** A Nexfar reporta timeout ao enviar pedidos.

    **Solução:** O timeout padrão da Nexfar é de **30 segundos**. Se o processamento do pedido demora mais, considere:
    - Aceitar o pedido de forma assíncrona (retorne imediatamente com `orderExternalId` e processe depois)
    - Otimizar o processamento no seu endpoint
  </Accordion>

  <Accordion title="Pedidos duplicados">
    **Problema:** O mesmo pedido aparece mais de uma vez no seu sistema.

    **Solução:** Seu endpoint **deve ser idempotente**. Use o campo `nexfarOrderId` do payload como chave única:
    - Antes de criar um novo pedido, verifique se já existe um com aquele `nexfarOrderId`
    - Se já existe, retorne o `orderExternalId` existente sem criar duplicata
  </Accordion>

  <Accordion title="Certificado SSL inválido">
    **Problema:** A Nexfar não consegue conectar ao seu endpoint.

    **Solução:** Certifique-se de que:
    - Seu endpoint usa HTTPS com certificado válido (não auto-assinado)
    - O certificado não está expirado
    - A cadeia de certificação está completa (incluindo certificados intermediários)
  </Accordion>

  <Accordion title="Assinatura HMAC não confere">
    **Problema:** A validação HMAC falha no seu endpoint.

    **Solução:** Verifique:
    - O shared secret está correto e é o mesmo nos dois lados
    - Você está calculando o HMAC sobre o **body raw** da requisição (antes de qualquer parsing)
    - O algoritmo está correto (padrão: `HmacSHA256`)
    - A comparação é feita corretamente (lembre-se do prefixo `sha256=`)
  </Accordion>

  <Accordion title="Retries constantes mesmo com endpoint funcionando">
    **Problema:** A Nexfar continua reenviando pedidos que já foram processados.

    **Solução:** Certifique-se de que sua response:
    - Retorna HTTP `2xx` (200, 201, etc.)
    - Contém o campo `orderExternalId` **preenchido** (não null, não vazio)
    - O Content-Type da response é `application/json`
  </Accordion>
</AccordionGroup>
